= Documentating Cleanup Audit — 2026-03-31
:description: Repository cleanup audit for the Smartresponsor documentating repository.

This note records the current cleanup situation for the `documentating` repository.

It is intentionally written as an audit and wave plan, not as a destructive cleanup patch.

== Scope

The repository is currently in a transition state rather than a single stable publication canon.

Three tensions are visible at once:

* legacy Markdown-first publication behavior,
* newer AsciiDoc-first publication behavior,
* and multiple generations of publish/build layers accumulated during active portal evolution.

Because of that, cleanup should be performed as a controlled refactoring of repository truth layers rather than as broad deletion.

== Confirmed Active and Transitional Layers

=== Legacy but still active owner-facing publish path

The workflow `.github/workflows/antora-gh-pages-owner.yml` still publishes from push on `master` and builds the site through `tools/build_antora_site_v4.py`.

That means it is not archival decoration. It is still part of the active publication surface and cannot be removed blindly.

=== Builder canon conflict

Two different build philosophies are visible.

The older `tools/build_antora_site.py` still describes Markdown as the editing surface during migration.

The newer `tools/build_antora_site_v4.py` already describes AsciiDoc as the canonical article source format and explicitly filters duplicates and noisy artifacts.

This means the repository does not yet expose one single canonical authored content policy.

=== Atlas workflow layering

The Quality Atlas / pre-RC atlas line introduced several generations in quick succession:

* `publish-v17-quality-atlas-after-v16`
* `publish-v18-pre-rc-quality-atlas-after-v17`
* `publish-v19-pre-rc-quality-atlas-inline-after-v17`

This is understandable during rapid evolution, but it is already a strong signal that the publication chain should later be collapsed into one final stable path.

=== Temporary trigger artifacts

The file `docs/v19-pre-rc-quality-atlas-trigger.txt` is a temporary publication trigger artifact.

This kind of file is acceptable during setup, but it should not remain a permanent resident once the final publish chain is stable.

== Cleanup Principles

=== Principle 1 — one authored content canon

The repository should converge toward one canonical authored publication format.

The strongest direction currently appears to be:

* authored article and portal content in AsciiDoc,
* generated portal tree derived from that canon,
* legacy Markdown either migrated, archived, or explicitly excluded.

=== Principle 2 — one final publish chain

After the pre-RC atlas chain is confirmed stable, the repository should converge toward one final publish workflow rather than multiple serial generations.

=== Principle 3 — no permanent trigger residue

Temporary trigger files may exist during setup, but they should not remain part of the permanent repository topology once the publication path is stable.

=== Principle 4 — cleanup should preserve meaning

The first cleanup waves should not rewrite article meaning or narrative substance.

The immediate task is topology cleanup:

* builder truth,
* workflow truth,
* authored source truth,
* generated vs temporary separation.

== Recommended Cleanup Waves

=== Wave A — stabilize the pre-RC atlas publish chain

Do not delete older workflow layers until the live pre-RC atlas publication path is confirmed stable.

=== Wave B — choose final builder canon

Decide which builder becomes the single canonical base for publication.

Likely candidates:

* evolve from `build_antora_site_v4.py`,
* or collapse the v17/v19 logic into one clearly named final builder.

=== Wave C — collapse workflow generations

Once stable, reduce the atlas-related workflow stack to a single final path and remove intermediate generations that no longer contribute meaningfully.

The strongest likely removal candidate from the currently visible stack is `publish-v18-pre-rc-quality-atlas-after-v17`.

=== Wave D — remove temporary trigger artifacts

Delete `docs/v19-pre-rc-quality-atlas-trigger.txt` after the publish chain is proven stable.

=== Wave E — article source cleanup

Perform a dedicated article inventory:

* canonical `.adoc`
* migrate-from-`.md`
* archive-only
* remove

This should be done carefully and file-by-file, especially where duplicate title/slug history exists.

== Immediate Recommendation

The next practical step should not be broad deletion.

The next practical step should be a targeted cleanup wave that does the following:

* selects the final pre-RC atlas publication path,
* marks intermediate atlas workflow generations for removal,
* removes temporary trigger residue,
* and launches a dedicated Markdown-to-AsciiDoc article inventory.

== Owner Note

This audit is intentionally conservative.

It prefers stable truth-layer cleanup over premature deletion while the publication chain is still settling.

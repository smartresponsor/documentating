= Accessing Report
:description: Current owner-oriented maturity report for the Accessing component.
:page-slug: accessing-report

This page is the current owner-oriented maturity report for *Accessing*.

It is written not as a defect log, but as a public-facing readiness portrait: what the component already demonstrates, what maturity band it currently occupies, and which leader-class direction it is converging toward.

== Component Portrait

Accessing is not shaped as a simple login screen or a lightweight security helper.

It reads as a Symfony-oriented account access and account security component: a system intended to manage registration, verification, MFA, password recovery, account locking, and operational discipline around the account lifecycle.

The component already shows signs of being treated as a governed subsystem rather than as a convenience layer. Its current shape suggests an internal access and identity surface designed for durability, structured evolution, and integration into a wider ecosystem.

== Ecosystem Role

Within the broader Smartresponsor direction, Accessing appears to occupy the role of an account access and account security component.

Its ecosystem role can be understood across several functions:

* as a registration and verification surface,
* as an MFA-aware account security layer,
* as a password recovery and reset surface,
* as a session and lock-discipline component,
* and as a governed account-core capability for the ecosystem.

This makes Accessing strategically more important than a typical login feature. It points toward a reusable internal access capability that other components can depend on.

== Current Maturity

Accessing already presents several strong signals of maturity:

* a clear account-centric product identity,
* explicit Symfony 8 and PHP 8.4 targeting,
* strong security package alignment,
* multiple test bands,
* a governed account model around verification, MFA, lock state, and failure tracking,
* and a local quality pipeline for style, static analysis, and tests.

In its current public reading, the component is no longer in a prototype-only band. It already behaves like a structured component moving toward release-candidate maturity.

== Leader Line

Accessing should not be measured against basic login forms or narrow authentication helpers.

Its natural comparison line is closer to the enterprise-class customer identity, account access, and governed authentication category, including the kind of capability bands represented by:

* registration and verification,
* password recovery and reset,
* MFA and second-factor enablement,
* session and lock discipline,
* and governed operational handling of account security.

This means the component is better understood as targeting the same strategic class of problem addressed by stronger account access and customer identity leaders, while expressing that class through a Symfony-native internal component.

In that sense, the leader line is not another login page.
It is closer to:

* enterprise account access class,
* enterprise account security class,
* enterprise governed identity-surface class,
* reinterpreted as an internal ecosystem component.

== Strength Profile

Several strengths are already visible and worth naming directly.

=== Account-centric model clarity

Accessing already carries a strong center of gravity around the Account entity itself.

That matters because many access systems become vague when registration, MFA, verification, recovery, and account security state are spread across disconnected technical surfaces.

That fragmentation is not the dominant signal here. The component instead signals a coherent account-centered model.

=== Security-aware domain design

The presence of email verification state, phone verification state, TOTP secret, second-factor enablement, lock state, and failed-login counters is one of the component’s strongest public signals.

These are not decorative fields. Together they show that Accessing already thinks in terms of governed account operations rather than only credential storage.

=== Operational access posture

The package surface already reflects the right account-runtime posture.

Mailer, notifier, rate limiting, verify-email, reset-password, CSRF, and Symfony Security together signal that the component is being shaped as a real access application rather than as a partial domain shell.

=== Verification-oriented quality contour

The quality contour is already coherent.

Accessing is not relying on ad hoc confidence. It already carries linting, cs checks, phpstan, phpunit, and a local full pipeline surface.

== RC Trajectory

The remaining path to a strong RC should be read not as a problem list, but as a maturity trajectory.

=== Enterprise parity expansion

The component is already structurally aligned with leader-class account access concerns. The next maturity band is about strengthening parity with broader enterprise expectations around user lifecycle breadth, administrative visibility, and advanced access controls.

=== Runtime surface strengthening

Accessing already has strong internal and modeled signals. The next maturity layer is about making the outward-facing access runtime and product shell feel as strong and complete as the internal account model.

=== Productized public posture

Internally, the component already looks more mature than many lightweight access layers. The next band is about translating that maturity into a clearer public-facing product posture that can be understood without reading the repository like an engineer.

== Public Verdict

Accessing is already well past the interesting draft stage.

It currently reads as a serious, Symfony-native account access and account security component with strong account-centered modeling, a coherent quality contour, and a clear operational direction around registration, verification, MFA, recovery, and lock discipline.

Its strongest present signal is not UI theater. Its strongest signal is substance: account lifecycle modeling, security-aware state, governed access operations, and a shape that clearly points toward durable internal identity and access use.

At this stage, Accessing can already be presented as a high-quality pre-RC component moving within an enterprise-class account access and identity-surface trajectory.
